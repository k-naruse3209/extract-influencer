# ADR-004: 認証・RBAC 設計

| 項目 | 内容 |
|------|------|
| ステータス | Accepted |
| 決定日 | 2026-03-20 |
| 決定者 | security-architect |
| 関連 ADR | ADR-001（技術スタック: JWT + httpOnly Cookie）、ADR-002（User.role: ADMIN/ANALYST/VIEWER） |

---

## コンテキスト

ADR-001 で JWT（短命アクセストークン）+ httpOnly Secure Cookie による認証方式が確定した。ADR-002 で User テーブルに `role` フィールド（ADMIN / ANALYST / VIEWER）が定義済みである。本 ADR では認証フローの詳細設計、RBAC の権限マトリクス、トークンリフレッシュ方針、セッション管理の考え方を定める。

---

## 決定

### 1. 認証方式: JWT + httpOnly Secure Cookie

**アクセストークン仕様**:

| 項目 | 値 |
|------|-----|
| アルゴリズム | HS256 |
| 有効期限 | 15 分 |
| 格納場所 | httpOnly + Secure + SameSite=Strict Cookie (`access_token`) |
| ペイロード | `{ sub: userId, role: UserRole, iat, exp }` |

**JWTペイロードにはPII（email, name）を含めない。** `sub`（userId = cuid）と `role` のみをペイロードに含める。これにより、トークンが万一漏洩した場合でも個人情報の露出を最小化する。

**採用理由**:

- **XSS 耐性**: httpOnly Cookie は JavaScript からアクセスできないため、XSS 攻撃でトークンを窃取されるリスクが低い。localStorage への保存は明示的に禁止する。
- **CSRF 緩和**: `SameSite=Strict` により、外部サイトからのリクエストに Cookie が付与されない。加えて、状態変更リクエスト（POST/PUT/DELETE）には Origin ヘッダーの検証を実施する。
- **ステートレス**: サーバー側にセッションストアを持たないため、水平スケール時にセッション同期の問題が発生しない。

### 2. リフレッシュトークン方針: 使用しない

**決定**: MVP ではリフレッシュトークンを実装しない。アクセストークン（15 分）の有効期限切れ時は再ログインを要求する。

**理由**:

1. **攻撃面の最小化**: リフレッシュトークンは長命（通常 7 日以上）であり、漏洩時の影響が大きい。リフレッシュトークンのローテーション実装を誤ると、トークン窃取 + リプレイ攻撃の窓が広がる。
2. **実装複雑度の削減**: リフレッシュトークンのローテーション、失効管理（Redis ブラックリスト）、同時セッション制御の実装が不要になる。MVP の開発速度を優先する。
3. **ユースケースの特性**: 本プラットフォームは社内ツール的な性質を持ち、長時間の連続セッションは主要ユースケースではない。15 分ごとの再ログインは許容範囲と判断する。
4. **将来の拡張余地**: リフレッシュトークンの追加は後方互換性を壊さずに実装可能。ユーザーからのフィードバックに基づいて Post-MVP で検討する。

**トレードオフ**: ユーザーは 15 分ごとに再ログインが必要。頻繁に使うユーザーにとっては摩擦になる可能性がある。この点は Post-MVP でリフレッシュトークン導入またはスライディングウィンドウ方式を検討する。

### 3. RBAC 権限マトリクス

| リソース / 操作 | ADMIN | ANALYST | VIEWER |
|----------------|-------|---------|--------|
| **ユーザー管理** | | | |
| ユーザー一覧取得 | OK | -- | -- |
| ユーザー作成・編集・削除 | OK | -- | -- |
| **API キー管理** | | | |
| API キー発行・取消 | OK | -- | -- |
| **インフルエンサー** | | | |
| 候補検索 | OK | OK | -- |
| 候補追加（URL/username 入力） | OK | OK | -- |
| 候補一覧閲覧 | OK | OK | OK |
| 候補削除 | OK | -- | -- |
| **スコアリング** | | | |
| スコア閲覧 | OK | OK | OK |
| スコア再計算トリガー | OK | OK | -- |
| **比較** | | | |
| 比較セッション作成 | OK | OK | -- |
| 比較セッション閲覧 | OK | OK | OK |
| **レポート** | | | |
| レポート生成（PDF/CSV） | OK | OK | -- |
| レポート閲覧・ダウンロード | OK | OK | OK |
| **監査ログ** | | | |
| 監査ログ閲覧 | OK | -- | -- |

**権限設計の原則**:

- VIEWER は読み取り専用。データの作成・変更・削除を一切行えない。
- ANALYST は分析業務に必要な操作（検索、追加、スコアリング、レポート生成、比較作成）を実行できる。ユーザー管理・API キー管理・候補削除は不可。
- ADMIN は全操作を実行できる。ユーザー管理・API キー管理を含む。

### 4. セッション管理の考え方

- **ステートレス**: サーバー側にセッション情報を保持しない。JWT の有効期限のみでセッションライフサイクルを管理する。
- **即時失効**: MVP では JWT の即時失効（ブラックリスト）を実装しない。パスワード変更時やアカウント無効化時は、最大 15 分のラグで旧トークンが有効となる。このリスクは短命トークン（15 分）により許容範囲と判断する。
- **ログアウト**: Cookie を削除する（クライアント側のトークン破棄）。サーバー側のブラックリストは MVP では不要。
- **同時セッション**: 制限しない。同一ユーザーが複数デバイス・ブラウザからログイン可能。
- **lastLoginAt の更新**: ログイン成功時に User.lastLoginAt を更新する。監査目的。

### 5. パスワードポリシー

- **ハッシュ**: bcryptjs（コストファクター 12）
- **最小長**: 8 文字
- **最大長**: 100 文字（bcrypt の 72 バイト制限を考慮しつつ、入力上限を設定）
- **複雑性要件**: MVP では最小長のみ。Post-MVP で大文字・数字・記号の要件を検討。

### 6. Cookie 設定

```
Name:     access_token
Value:    <JWT>
HttpOnly: true
Secure:   true (本番環境)
SameSite: Strict
Path:     /
MaxAge:   900 (15分 = 900秒)
```

開発環境（localhost）では `Secure: false` を許容する（HTTP での開発を可能にするため）。`NODE_ENV` で切り替える。

## 脅威分析

| 脅威 (STRIDE) | 対策 |
|---------------|------|
| Spoofing (なりすまし) | bcryptjs によるパスワードハッシュ、JWT 署名検証 |
| Tampering (改ざん) | JWT の HMAC 署名、httpOnly Cookie |
| Repudiation (否認) | AuditLog テーブルによる操作記録 |
| Information Disclosure (情報漏洩) | JWT ペイロードの PII 最小化、パスワードフィールドの応答除外 |
| Denial of Service (DoS) | ログイン試行のレート制限（Post-MVP） |
| Elevation of Privilege (権限昇格) | RolesGuard による全エンドポイントの権限検証 |

---

## 保留事項

- ログイン試行のレート制限（ブルートフォース対策）は Post-MVP で実装する
- リフレッシュトークンの導入は、ユーザーフィードバックに基づいて Post-MVP で検討する
- MFA（多要素認証）は Post-MVP で検討する
- JWT の即時失効（Redis ブラックリスト）は、アカウント無効化の即時性が求められた場合に導入する
