# リリース準備 スキル

## 目的

本番デプロイ前の最終確認を体系化し、「デプロイしたら壊れた」を防ぐ。

---

## リリース前チェックリスト

### コード品質

- [ ] ユニットテスト ≥ 80%（スコアリング ≥ 90%）グリーン
- [ ] 統合テスト全件グリーン
- [ ] E2E 主要フロー全件グリーン（`e2e-checklist.md` 参照）
- [ ] `console.log` / `print` デバッグが残っていない
- [ ] `TODO` / `HACK` がコミットに含まれていない
- [ ] TypeScript `any` が新規追加されていない

### セキュリティ

- [ ] security-reviewer のレビュー承認済み
- [ ] `.env` / secrets がコミットに含まれていない
- [ ] RBAC テスト全ロール通過済み
- [ ] audit log が動作確認済み

### DB・マイグレーション

- [ ] マイグレーションファイルが用意されている
- [ ] rollback スクリプトが用意されている（必ずセット）
- [ ] マイグレーション後にテストが通ることを staging で確認済み
- [ ] data-architect のレビュー承認済み（スキーマ変更時）

### 観測可能性

- [ ] エラーログが適切に出力される
- [ ] API p95 レスポンスタイムが 500ms 以内
- [ ] アラート設定が正しく動作する

### ドキュメント

- [ ] API ドキュメントが更新済み
- [ ] `docs/agent-system/` の関連ドキュメントが更新済み
- [ ] CHANGELOG または PR 説明に変更内容が記載済み

---

## Smoke テスト手順

デプロイ後に本番（または production-like 環境）で実施する:

1. ヘルスチェックエンドポイントが 200 を返す
2. Instagram アカウント検索が動作する（テスト用 username で確認）
3. PDF が生成・ダウンロードできる
4. CSV エクスポートができる
5. 管理画面にログインできる

---

## ロールバック判断基準

以下のいずれかが発生したら即座にロールバックを検討する:
- エラーレートが通常の3倍以上
- p95 レスポンスタイムが 1000ms を超える
- 主要フロー（検索・PDF出力）が動作しない
- セキュリティ脆弱性が発見された

---

## Feature Flag 運用

- 大きな機能変更は feature flag で制御する
- flag 名は `ff_{feature_name}` の形式（例: `ff_tiktok_manual_input`）
- リリース後、安定稼動を確認してから flag を削除する（残存させない）

---

## gotchas

- マイグレーションを忘れてデプロイするとアプリが起動しない。必ず staging で事前確認する
- ロールバックスクリプトを本番初日に作るのは遅い。実装時に必ずセットで用意する
- smoke テストはデプロイ後 10 分以内に実施する。問題があれば早期に検知できる
