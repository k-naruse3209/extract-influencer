# Architecture Decision Record (ADR)

## 目的

重要な技術的決定を記録し、「なぜその設計にしたか」を後から参照・変更できるようにする。

## ADR を書くべき決定

- 技術スタックの選定（フレームワーク・DB・インフラ）
- 認証方式の選定
- キャッシュ戦略
- スコアリングアーキテクチャ
- データ分離方式（事実/推定/LLM生成の保存方式）
- Instagram API 連携方式
- PDF/CSV 生成方式

## ADR の保存場所

`docs/agent-system/adr/adr-NNN-[タイトル].md`

番号は 3 桁で付番（001, 002, ...）

## テンプレート

`templates/adr-template.md` を使用する。

## ADR 作成手順

1. 問題を定義する（なぜ決定が必要か）
2. 検討した選択肢を列挙する（最低 2 案）
3. 各選択肢のトレードオフを評価する
4. 決定を記録する（採用案 + 理由）
5. 結果として生じる影響を記録する
6. 保留事項があれば記録する

## ステータス管理

- `Proposed`: 提案中
- `Accepted`: 採用
- `Deprecated`: 廃止（代替 ADR を参照）
- `Superseded`: 上書きされた（新 ADR を参照）

## gotchas

- ADR は「過去の判断」を記録するもの。変更する場合は新しい ADR で上書きする
- 「とりあえず後で書く」と書かれないままになる。決定したらその場で書く
- 代替案を書かない ADR は価値が半減する（なぜ他を選ばなかったか が重要）
