# テスト・検証 スキル

## 目的

このプロジェクトの品質基準を満たすテストを計画・実施し、「動いているように見えるが実は壊れている」状態を防ぐ。

---

## カバレッジ基準

| 対象 | 最低基準 |
|------|--------|
| ユニットテスト（全体） | ≥ 80% |
| スコアリングロジック | ≥ 90% |
| 統合テスト | API エンドポイント全件 |
| E2E | 主要フロー全件 |

---

## テスト種別と実施方針

### ユニットテスト

- スコアリングロジック・データ変換・集計関数を優先する
- 外部 API 呼び出しはモックを使う
- データ変換・集計ロジックはモックしない
- テストデータは `tests/fixtures/` のフィクスチャを使う（実際のユーザーデータ禁止）

```
// 命名規則
describe('ScoreCalculator', () => {
  it('returns unavailable when followerCount is missing', () => { ... })
  it('calculates engagementRate correctly with 30 posts', () => { ... })
  it('normalizes score to 0-100 range', () => { ... })
})
```

### 統合テスト

- API エンドポイント全件に対して実施する
- テスト用 DB を使う（本番 DB に接続しない）
- Instagram API は sandbox / モックを使う（本番 API を直接叩かない）

### E2E テスト

主要フロー（最低限これは通す）:
1. 候補検索 → 絞り込み → 候補選択
2. 候補を保存 → 保存済み一覧に反映
3. 比較（最大10件）→ 比較画面表示
4. PDF レポート生成 → ダウンロード
5. CSV エクスポート → 文字コード確認（UTF-8 BOM）

---

## 検証観点（このプロジェクト固有）

| 観点 | 確認内容 |
|------|---------|
| スコア範囲 | 0〜100 の範囲内か |
| 欠損対応 | unavailable が正しく返るか（null や 0 になっていないか） |
| rate limit | Instagram API rate limit 超過時にリトライ/エラーを正しく返すか |
| PDF必須フィールド | 全必須フィールドが PDF に含まれるか |
| CSV文字コード | UTF-8 BOM 付きで出力されるか（Excel対応） |
| 推定値表示 | estimated フィールドに ※ マークが表示されるか |
| LLM JSON崩れ | プロンプト返却値が JSON 崩れしたときフォールバックが動くか |

---

## 回帰テスト（スコアリング変更時必須）

1. 変更前のスコアをサンプルデータで全件記録する
2. 変更後に同じデータで再計算する
3. 差分が ±5 点を超えるケースを全件レビューする
4. 意図的な変更でなければロールバックを検討する

## gotchas

- スコアリングロジックを変更したら必ず既存スコアとの回帰テストを実施する
- E2E テストは Instagram API のサンドボックスを使う（本番 API を E2E から叩かない）
- PDF の検証は「ファイルが生成された」だけでなく「必須フィールドが入っているか」まで確認する
- CSV の UTF-8 BOM 確認は Excel で開いて文字化けしないことを目視確認する
