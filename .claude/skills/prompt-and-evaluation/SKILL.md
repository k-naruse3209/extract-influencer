# プロンプト設計・評価 スキル

## 目的

LLM（Claude）を使ったインフルエンサー分析機能のプロンプトを、品質・安全性・再現性を保ちながら設計・管理・改善する。

---

## プロンプト管理方針

- プロンプトはコード内の定数ファイルで管理する（DBに持たせない）
- バージョンは `v1`, `v2` ... で命名し、変更時は古いバージョンをコメントアウトで残す
- 変更時は eval ケースを実行してから本番適用する
- モデル名をプロンプトファイルに記録する（モデル変更時に参照できるように）

---

## ブランド適合コメント生成プロンプト設計原則

```
1. 役割を明示する（You are an influencer marketing analyst...）
2. 入力データの型を明確にする（事実/推定/欠損を区別して渡す）
3. 出力フォーマットを JSON で強制する
4. 判断根拠を必ず含めるよう指示する
5. 推定・不確実な場合は「〜と考えられる」「〜の可能性がある」と書かせる
6. 断定形（「このインフルエンサーは必ず〜」）を禁止する
```

### テンプレート → `templates/prompt-template.md` を参照

---

## リスクコメント生成プロンプト設計原則

- 炎上リスク・景品表示法・薬機法の観点を明示する
- 過去投稿テキストのサンプルを渡す（最大15件）
- LLM に「どのキーワードがリスクと判断したか」を出力させる
- スコアは LLM に出力させない（ルールベースで算出し、コメントのみ LLM に任せる）

---

## 幻覚抑制対策

1. **根拠なき断定を禁止するシステムプロンプト**を必ず入れる
2. **入力データに含まれない情報の推測を禁止**する文言を入れる
3. **出力 JSON のスキーマを厳密に定義**し、不要フィールドを生成させない
4. **JSON parse が失敗した場合のフォールバック**を必ず実装する（try/catch）
5. **eval ケースで幻覚パターンを定期チェック**する

---

## eval ケースの設計方針

- 最低10ケースを用意する（正常系5 + 異常系3 + エッジケース2）
- 期待出力は「完全一致」でなく「含むべきキーワード + 含んではいけないキーワード」で評価する
- 評価指標: 根拠の有無・断定の有無・JSON形式の正確性・スコア範囲
- `templates/eval-case-template.md` を使う

---

## gotchas

- JSON出力を強制しても崩れることがある。必ず try/catch でラップする
- モデルを変更するとプロンプトの効き方が変わる。モデル変更 = eval 再実行を必須とする
- 長い投稿テキストはトークン制限に引っかかる。事前にトランケートする（最大2000トークン/投稿）
- 日本語と英語が混在する投稿に注意。言語を明示して渡す
