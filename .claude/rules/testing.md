# テストルール

## カバレッジ基準

- ユニットテスト: ≥ 80%（ビジネスロジック・スコアリング層は ≥ 90%）
- 統合テスト: API エンドポイント全件
- E2E: 候補検索 → 保存 → 比較 → レポート出力 の主要フロー

## テストを書く基準

- スコアリングロジックは必ずユニットテストを書く（変更時の回帰検知が必須）
- 外部 API（Instagram Graph API 等）の呼び出しはモックを使う
- ただしデータ変換・集計ロジックはモックしない
- DB 操作は統合テスト（テスト用 DB を使う）

## テストデータ

- テストデータはコード内のフィクスチャとして管理する
- 実際のインフルエンサーデータをテストデータに使わない（PII 問題）
- 架空のサンプルデータを `tests/fixtures/` に用意する

## テストの命名

- `describe`: テスト対象（例: `ScoreCalculator`）
- `it` / `test`: 「何をすると何になるか」（例: `returns low confidence when follower count is unavailable`）

## CI での自動実行

- PR 作成時にユニットテスト + 統合テストを自動実行する
- E2E はステージング環境へのデプロイ後に実行する
- テスト失敗時はマージブロックとする

## 検証すべき観点（このプロジェクト固有）

- スコアが 0〜100 の範囲内か
- 欠損データ時にスコアが `null` でなく適切な値（または unavailable）を返すか
- Instagram API の rate limit 超過時に適切にリトライ・エラーを返すか
- PDF 出力時にすべての必須フィールドが含まれているか
- CSV エクスポート時に文字コードが UTF-8 BOM 付きか（Excel 対応）

## gotchas

- スコアリングロジックを変更したら必ず既存のスコアとの回帰テストを実施する
- E2E テストは Instagram API のサンドボックスを使う（本番 API を E2E から叩かない）
